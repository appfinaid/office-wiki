Select the ID as indicated above, and click save." And then the next screenshot has a screen, and it's got the load circled.

Jennifer Buckles: So, here's the issue with why that's a question mark. What load is doing, so when you match something, you're just connecting this ISIR with this student, but nothing is actually attached in Renana. There's no record created in Renana. There's a job that has to run that creates that record, and that's when it stamps it with the ISIR was received on this date, and then that stamp date is the date that the data load process uses when it's picking up new ISIRs to track C flags and comment codes and all of that stuff.

So if you load individually, and then for whatever reason, the ISIR process doesn't run tonight, the file didn't come in, something crazy happens with Automic, and it just doesn't run, that ISIR that you loaded today does not get run through all those processes to track the C flags, to track the whatever comment flags, whatever comment codes, whatever tracking codes are required. It misses all of that. Then tomorrow, when that process runs, it's picking up tomorrow's date. Well, if it got loaded today, it's never gonna get picked up. So, when you load, and then the ISIR process doesn't run, you could have a disconnect between your tracking elements.

So what we did to remedy that is we have a special load that runs before the ISIR load goes. So, I would say, no, you do not want anybody running load because of the potential for it not to get tracking updated. Maybe you just need to update the directions.

The advantage of being able to do load when you're doing duplicates is you can only—so when you run the match, it's gonna then put both duplicates back in duplicate unless one is loaded. So, if you got to your two duplicates, and let's say you found a match, you can only match one at a time, you can't match both of them. So you match one, that one has to load, then when the match runs again, it'll match the other one automatically, because now it's the only one out there. But that's gonna take the night of it running in order for the second one to actually be able to load.

So let me just talk through that. You have a duplicate that has a match. You match the first one, you just match it, you don't load it, you just match it. Move on to all the rest of your duplicates. Tonight, when the data load runs, the first job that runs is a load job, so it picks up that match you already did and loads it. Then it goes through the whole ISIR process, which is look for new matches, then it'll match the second one, and load those. So it'll load both of them tonight. Then the newest one will get run through all the tracking and all of that, but both would be matched tonight. But you'd have to wait for tonight, so that other one is still going to be hanging out there in your duplicate list until the ISIR process runs tonight. So the only advantage of being able to load on demand is that you could load it, rerun the match, go ahead and match the second one and load it right then. But, if the ISIR process doesn't run tonight, you've just missed all the tracking. So, I feel like the advantage of having it loaded right now versus the risk of missing tracking updates, I feel like it would be better to do that.

Now, there may be a case when you've got an urgent, "we've got to get this thing loaded, we know what the comment codes are," and so maybe special people, like you all, would have that option, to where you could load it if you had to, but you know the implications of, "if I'm loading this, I need to make sure all the tracking's updated, just in case something happens with the data load tonight." But I think the more people you have doing that, the more risk of it missing the tracking and the data load.
