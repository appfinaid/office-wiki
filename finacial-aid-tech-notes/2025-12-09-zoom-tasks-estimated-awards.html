Financial Aid Estimated Awards Process: FASH Grouping and Packaging Workflow

Current Processes and Procedures

The team is currently in the preparatory phase for the 2026-2027 (26-27) academic year estimated awards cycle.

  • ISIR Loading: The system is currently running the basic ISIR load processes (RTP, RCPTP).
  • Disabled Processes: Several downstream processes are currently disabled or have not yet been run for the 26-27 aid year, including:
    • Fixate periods.
    • Fixed year in college.
    • Satisfactory Academic Progress (SAP).
    • Data loads.
    • Packaging Group Assignments.
  • Scholarship Management (FASH): The institutional scholarship process involves mass uploading awards (over 7,000 records) or using the Banner portal. This process relies on the underlying Banner "Packaging Group" to function efficiently.
  • Budgeting: Budgets are mostly finalized, with the exception of the final freshman amount. However, the system allows for budgeting to update continuously until a package is locked.

Identified Problems

1. Manual Processing Overhead in FASH

The most critical immediate issue is a workflow inefficiency regarding the FASH (Fund Awarding Scholarship) process.

  • The Issue: FASH relies on students having an assigned "Packaging Group" in Banner to post awards automatically. Because the "Grouping" process has not been run for the 26-27 aid year, students currently have no group assigned.
  • The Consequence: If the mass upload of 7,000+ scholarships is attempted without existing groups, the Financial Aid staff (specifically Lindley) would be forced to manually add a packaging group to every single student record. This is a significant waste of time and introduces high potential for human error.

2. Packaging Sequencing and Over-Awarding

  • The Issue: There is a risk of inefficient workflow if "Estimated Packaging" is run before institutional scholarships are posted.
  • The Consequence: If a student is packaged with need-based aid first, and then a scholarship is added later, the student may become "over-budget" or "over-awarded." This requires staff to manually "work overbudgets" to reduce other aid.

3. "Standard Group" Chain Failure Risk

  • The Issue: The "Standard Group" process (for students without a FAFSA) was previously part of the main ISIR load chain.
  • The Risk: If the population selection (pop sel) for this group pulls zero records, it causes the entire ISIR load chain to fail and abort.

Proposed Solutions and New Workflows

To resolve the manual processing bottleneck and ensure accurate packaging, the team agreed to the following solutions:

1. Pre-Run Grouping (Immediate Fix)

The technical team will run the Grouping process for all 26-27 students before the scholarship upload occurs. This includes both:

  • ISIR Groups: Students with a valid FAFSA.
  • Standard Groups: Students without a FAFSA (assigned a generic group).
  • Outcome: This ensures that when FASH is run, the system recognizes the students' groups and posts awards automatically, eliminating the need for Lindley to manually touch 7,000 records.

2. Workflow Sequencing: Scholarships First

The workflow will be strictly sequenced to minimize rework:

  1. Group Assignment: Run grouping for all students.
  2. FASH Upload: Post institutional scholarships.
  3. Estimated Packaging: Run packaging after scholarships are on the record.
  • Outcome: The packaging engine will "see" the scholarship as an existing resource and calculate remaining need/eligibility correctly the first time, preventing over-award situations.

3. Separation of AppWorx Chains

The "Standard Group" process has been isolated into its own execution chain. It is no longer dependent on the ISIR load chain. This mitigates the risk of the main data load failing if no students qualify for the standard group on a specific day.
